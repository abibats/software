## 第 2 页：项目概述

**标题：** 项目背景、目标与开发模式

**正文要点：**

**项目背景**
- 高校自习室座位资源紧张，人工占座、恶意占座现象普遍
- 亟需一套在线预约、签到核验、违约处理的数字化管理平台

**整体目标**
- 为学生提供座位查询、预约、签到、取消等全流程服务
- 为管理员提供自习室/座位管理、参数配置、角色权限与数据统计能力
- 实现自动违约处理与智能助手问答，提升资源利用率与管理效率

**开发模式**
- 采用 **Scrum 敏捷开发**：3 次迭代 + 1 周部署汇报期
- 需求驱动：**用户故事 → 任务分解 → 验收标准 → 自动化测试**
- 协作模式：6 人 Feature Branch + Pull Request + Code Review
- 工程实践：GitHub Actions CI/CD + 华为云 ECS 自动部署 + DevCloud 项目管理

**技术选型**
- 前端：原生 HTML / CSS / JavaScript（无框架）
- 后端：Python 标准库 HTTP Server + REST API
- 数据层：SQLite（7 张业务表）
- 部署：华为云 ECS + Nginx 反向代理 + systemd 服务管理

## 第 10 页：CI/CD 与 DevCloud 开发流程（Git 与 Commit）

**标题：** 分支管理与规范提交

**正文要点：**

**分支策略：Feature Branch Workflow**
- `main` 为受保护主分支，禁止直接 push，仅通过 PR 合并
- 功能分支按职责命名，共 10+ 条活跃分支：

| 分支名 | 职责 |
|--------|------|
| feature/student-frontend | 学生端前端 |
| feature/admin-frontend | 管理端前端 |
| feature/reservation-backend | 预约业务后端 |
| feature/auth-backend | 权限与数据库 |
| feature/testing-devops | 测试与 DevOps |
| feature/bugfix-core-logic | 核心业务修复 |
| assistant-api-integration | 智能助手 API |

**PR 合并记录：** 共合并 13 个 Pull Request（#2 ~ #13），每个 PR 至少 1 位成员 Review

**Conventional Commits 规范**
```
<type>(<scope>): <description>
```
- `feat`：新功能（如 `feat(reservation): add seat conflict detection`）
- `fix`：缺陷修复（如 `fix(checkin): validate room daily code correctly`）
- `test`：测试补充（如 `test(backend): add reservation overlap test`）
- `docs` / `ci` / `refactor`：文档、流水线、重构

**原子提交原则：** 每个 commit 对应一个明确任务，便于追溯与回滚

## 第 11 页：CI/CD 与 DevCloud 开发流程（AI 增量开发）

**标题：** AI 辅助增量开发与 DevCloud 项目管理

**正文要点：**

**AI 增量开发使用方式**
1. **任务粒度控制：** 每次仅描述一个明确子任务（如"在 POST /api/reservations 中添加时间冲突检测"），避免一次性生成大量不可控代码
2. **上下文精准提供：** 指明文件路径、函数名、期望行为与现有架构约束
3. **逐次迭代推进：** 先实现功能 → 再修复缺陷 → 最后补充测试与文档
4. **人工 Review 把关：** 理解 AI 生成代码的每一行，确保符合 RBAC 架构与无框架技术路线

**典型 AI 辅助开发示例**
- 任务 1："实现预约时间冲突检测" → 原子 commit → 对应 test_reservation_rejects_overlap_for_same_seat
- 任务 2："修复 auto_expire 读取系统参数错误" → 精准 fix commit
- 任务 3："补充取消预约边界测试" → test commit

**华为 DevCloud 项目管理**
- Scrum 项目创建与成员管理（梁志杰）
- 工作项管理：用户故事与 Bug 追踪
- 代码仓库：镜像 GitHub 主仓库
- 构建与部署任务：配合流水线截图归档
- 测试管理：手工测试用例补充记录

**问题跟踪处理流程**
1. **发现：** 例会讨论 / Code Review / CI 失败 / 联调测试
2. **记录：** 写入 Review 意见或 Issue，标注严重级别
3. **分配：** 指定负责人与修复截止日期
4. **修复：** 在 feature 分支提交 fix，本地测试通过后提 PR
5. **验证：** Reviewer 复审 + CI 自动测试通过
6. **关闭：** 合并 main，更新文档，下次例会确认

## 第 14 页：流水线与自动部署（CI/CD 五阶段）

**标题：** 代码检查、构建、测试、部署流水线

**正文要点：**

**GitHub Actions 五阶段流水线**（`.github/workflows/ci.yml`）

```
push/PR → Lint → Build → Test → Deploy → Smoke Test
```

| 阶段 | Job 名称 | 执行内容 | 触发条件 |
|------|----------|----------|----------|
| 1. 代码检查 | Code quality check | flake8 静态检查 backend 核心文件 | 每次 push/PR |
| 2. 构建验证 | Build and verify | Python 语法编译 + 前端资源存在性 + 模块加载 | 每次 push/PR |
| 3. 自动化测试 | Automated tests | 33 个 unittest 用例（Windows 环境） | lint + build 通过后 |
| 4. 自动部署 | Deploy to Huawei Cloud | SSH 到 ECS → git pull → systemctl restart | 仅 main 分支 push |
| 5. 冒烟测试 | Post-deploy smoke test | 线上登录、用户信息、统计、座位列表验证 | deploy 成功后 |

**部署架构**
```
浏览器 → Nginx(:80) → Python HTTP Server(:8000) → SQLite
         华为云 ECS (Ubuntu 24.04, 2vCPU 2GiB)
         systemd 管理 study-seat 服务
```

## 第 16 页：总结

**标题：** 项目过程总结、实践收获与改进方向

**正文要点：**

**项目过程总结**
- 以用户故事为需求入口，验收标准直接映射自动化测试，形成完整需求—开发—验证闭环
- Feature Branch + PR + Code Review 机制有效保障代码质量与知识共享
- 三次迭代由框架到核心业务再到测试部署，进度可控、交付可度量
- CI/CD 五阶段流水线实现从代码提交到线上部署的全流程自动化
- 例会制度与问题跟踪流程确保团队沟通顺畅、缺陷及时闭环

**实践收获**
- 深入理解敏捷 Scrum 在课程项目中的落地方式（故事点、迭代、Review）
- 掌握无框架条件下的 HTTP 协议、REST API 设计与 RBAC 权限建模
- 实践 Conventional Commits、原子提交与可追溯的工程协作规范
- 体会 AI 辅助开发需"精准引导、增量迭代、人工把关"，而非一次性生成
- 完成从本地开发到华为云生产部署的完整 DevOps 实践

**现存问题与改进方向**

| 现存问题 | 改进方向 |
|----------|----------|
| 无前端自动化测试（E2E） | 引入 Playwright 等端到端测试框架 |
| 智能助手依赖外部 API，离线能力有限 | 增强本地规则引擎覆盖率 |
| SQLite 不适合高并发场景 | 生产环境迁移至 PostgreSQL/MySQL |
| 缺少连续违约惩罚机制 | 扩展 US-3.x 违规处理策略 |
| DevCloud 与 GitHub 双平台管理 | 统一至单一 DevOps 平台减少同步成本 |
